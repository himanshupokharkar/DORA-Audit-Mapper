DORA AUDIT MAPPER — REUSABLE USER PROMPTS (V3)
═══════════════════════════════════════════════════════════════════════

This file contains four reusable prompts for use with the DORA Audit
Mapper system prompt. Copy the prompt you need into a new chat, fill
in the bracketed placeholders, and send.

Companion system prompt: DORA_Audit_Mapper_System_Prompt_v8_PUBLIC.txt

CHANGE FROM V2: Audit objective is now optional. You do not need to
provide it to get a full regulation mapping. If you provide it, it
tailors the Coverage Assessment section only — the five-section
mapping is identical regardless of objective.

Knowledge base files this is designed to work with:
  1. Main DORA Regulation (EU 2022/2554)
  2. RTS on ICT Risk Management Framework (EU 2024/1774)
  3. RTS on Threat-Led Penetration Testing (EU 2025/1190)
  4. RTS on Incident Classification (EU 2024/1772)
  5. RTS on Incident Reporting (EU 2025/301)

═══════════════════════════════════════════════════════════════════════
PROMPT 1 — STANDARD MAPPING (MOST COMMON)
═══════════════════════════════════════════════════════════════════════

USE WHEN: You want a full structured mapping of all applicable DORA
articles and RTS provisions for a specific audit topic and entity.
Audit objective is optional — include it if you want the Coverage
Assessment tailored to your testing approach.

────────────────────────────────────────────────────────────────────
I want to map DORA obligations for an audit. The details are:

- Audit topic: [SPECIFY — e.g. ICT incident classification and reporting]
- Entity type: [SPECIFY — e.g. non-microenterprise credit institution,
  not designated as significant under SSM]
- Scope exclusions: [SPECIFY any, or state "none"]

Please return the full structured five-section mapping with Article
Text Appendix.
────────────────────────────────────────────────────────────────────

OPTIONAL: Add this line if you want the Coverage Assessment tailored
to a specific audit approach:
- Audit objective: [design effectiveness / operating effectiveness /
  compliance review / combination]


═══════════════════════════════════════════════════════════════════════
PROMPT 2 — TARGETED ARTICLE CHECK
═══════════════════════════════════════════════════════════════════════

USE WHEN: You want a focused list of only the directly applicable
articles, without the full five-section structure.

────────────────────────────────────────────────────────────────────
For an audit of [TOPIC] at a [ENTITY TYPE], which DORA articles and
RTS provisions are directly applicable?

Exclude:
  - ESA empowerment provisions
  - Competent authority and Lead Overseer procedural duties
  - Definitions
  - Permissive provisions

Return only the directly applicable provisions and any conditional
obligations with their triggers clearly stated.
────────────────────────────────────────────────────────────────────


═══════════════════════════════════════════════════════════════════════
PROMPT 3 — SANITY CHECK EXISTING AUDIT PLAN
═══════════════════════════════════════════════════════════════════════

USE WHEN: You have already identified articles for an audit and want
a second opinion on completeness and accuracy.

────────────────────────────────────────────────────────────────────
I have already identified the following DORA and RTS provisions as
in scope for my audit:

[PASTE YOUR CURRENT LIST OF ARTICLES HERE]

My audit context is:
  - Audit topic: [SPECIFY]
  - Entity type: [SPECIFY]

Please tell me:
  1. Are there additional articles or RTS provisions I should add?
  2. Are any of my listed articles not actually applicable to this
     entity type or scope?
  3. Are there any conditional obligations whose triggers I should
     evaluate?
  4. Any cross-references between my listed articles and the loaded
     RTS documents that I might be missing?
────────────────────────────────────────────────────────────────────


═══════════════════════════════════════════════════════════════════════
PROMPT 4 — CROSS-DOCUMENT COMPREHENSIVE MAPPING
═══════════════════════════════════════════════════════════════════════

USE WHEN: Your audit topic spans multiple DORA instruments and you
want explicit cross-document linkage surfaced. Best used for incident
management (spans Main DORA + RTS Incident Classification + RTS
Incident Reporting), third-party risk (spans Main DORA + RTS ICT-RMF
+ unloaded RTS on Subcontracting), or TLPT (spans Main DORA + RTS
TLPT).

────────────────────────────────────────────────────────────────────
I need a comprehensive mapping for an audit topic that spans multiple
DORA instruments. The details are:

- Audit topic: [SPECIFY — e.g. "incident management, classification,
  and reporting end-to-end"]
- Entity type: [SPECIFY]
- Scope exclusions: [SPECIFY any, or state "none"]

I expect this mapping to draw from Main DORA AND multiple RTS
documents. Please:

1. Surface all cross-document linkages explicitly in Section 3
2. Use the "Expanded by" column to show every linked instrument
   regardless of whether it is loaded or not
3. In Coverage Assessment, list any "(not yet loaded)" instruments
   the auditor should obtain next
4. Flag if any obligation requires multiple instruments read together
   to interpret correctly
────────────────────────────────────────────────────────────────────


═══════════════════════════════════════════════════════════════════════
TIPS FOR USING THESE PROMPTS
═══════════════════════════════════════════════════════════════════════

1. REPLACE BRACKETED PLACEHOLDERS with your actual audit details.
   Delete the brackets and the reminder text once filled in.

2. BE SPECIFIC ABOUT ENTITY TYPE. "Non-microenterprise credit
   institution, not designated as significant under SSM" gives much
   better results than "bank". Key distinctions the system acts on:
   - Microenterprise vs non-microenterprise (DORA Art.16(1))
   - Standard investment firm vs small and non-interconnected
   - Significant credit institution vs non-significant
   - TLPT-designated entity vs non-designated
   - Insurance undertaking vs credit institution (sectoral rules)

3. AUDIT OBJECTIVE IS OPTIONAL. The system will produce a full
   regulation mapping without it. Only include it if you want the
   Coverage Assessment to contain testing-approach-specific guidance
   (e.g. for operating effectiveness, what evidence to gather; for
   compliance review, what to attest row by row).

4. THE ARTICLE TEXT APPENDIX IS AUTOMATIC. Every mapping output ends
   with the full verbatim text of every article cited in Sections 1-4.
   You do not need to request this separately.

5. CONVERTING TO EXCEL. To turn any mapping output into an Excel
   testing workbook, use the separate DORA_Excel_Conversion_Prompt.txt
   in this folder. Copy the full mapping output (including the Article
   Text Appendix) and paste it into that prompt in a new chat.

6. THE SYSTEM NEVER INVENTS OBLIGATIONS. If a relevant article seems
   missing, check whether the source document is loaded in the
   knowledge base. The Coverage Assessment will list any instruments
   referenced but not yet loaded.

7. FOR HIGH-STAKES AUDITS. Run Prompt 1 first to get the full
   picture, then Prompt 3 against your refined list to catch gaps
   before finalising scope.

8. PROMPT 4 IS BEST FOR. Incident management (three loaded files
   intersect), TLPT (deep single-RTS coverage plus DORA Chapter IV),
   and third-party risk (DORA Chapter V plus RTS cross-references).
